# Build Synergy Posture

## Purpose

This note explains how `aoa-playbooks` may describe build synergy and narrow specialization without mutating runtime state or upstream source meaning.

## Core rule

Synergy is scenario-shaped.

A party template may say that a route works best when:
- certain abilities are loaded together
- certain feats reinforce the route
- a reviewer or evaluator slot is always present
- one slot remains gated until stronger proof appears

That is composition posture, not metaphysical destiny.

## Narrow specialization

Use specialization when a scenario truly benefits from a tighter build lane.
Examples:
- proof-first bounded change pair
- validation/remediation triad
- incident routing stabilizer
- restartable inquiry loop with recall guardrails

A specialization should stay local to the scenario family unless broader evidence justifies promotion.

## Allowed loadout language

Good first bridge-wave loadout language:
- required abilities
- optional abilities
- preferred feats
- optional artifact classes
- minimum reputation slices when trust posture matters
- explicit cautions about adapter, overlay, or local-runtime posture

## Not allowed here

Do not put these inside playbook build posture:
- live tool bindings
- session budgets
- transport protocol details
- hidden reward modifiers
- per-run item drops or economy loops
- secret route logic the source playbook does not expose

## Final rule

A playbook build is a reviewed lane sketch.
It should help a future frontend or orchestrator compose a party without pretending the sketch is already the live run.
